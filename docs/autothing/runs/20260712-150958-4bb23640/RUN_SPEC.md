# RUN_SPEC — Ekoa Apps Get an Operator (run 20260712-150958-4bb23640)

Derived from `BRIEF.md` (LEDGER v0.2) + planning exploration on 2026-07-12. Status: DRAFT until FLOW_PLAN derives from it; assumptions ledger is the operator's post-run review surface.

## What / why

Every Ekoa-built **app** ships a dedicated operator assistant bound to that app: it automates the app by driving its UI in-page (never fighting the user), teaches it (zero-token declarative tours + coached modes), and answers the app's domain from org-scoped indexed knowledge with citations. Admin-gated "change the app" and the roles/identity/request-changes system land in a contiguous security block at the END of the run (model-tier sequencing rule). Commercially, every end user becomes a metered token consumer; tours and registry-only actions are free.

## Ground truth found in exploration (what already exists)

- **Bases exist, unwired.** `api/assets/bases/{landing,document,presentation,app-auth-persistent,app-integration-heavy}` with manifests, tokens.json, instructions/, scaffold/, wiring/, recipes/, skills/ + `CSS_VARS_CONTRACT.md` — NOTHING in `api/src` reads them. `apps/scaffold.ts` already accepts `templateScaffoldFiles` (unused seam) and reads generic starters from `api/assets/scaffold-templates/`.
- **Tour stack survives end-to-end.** Declarative demo-spec (`api/assets/demos/_schema.json`, ~30 authored legal tours), registry + validator (`api/src/services/demo-registry.ts`), injected in-app bridge (`api/assets/demo-bridge-client.js`, served at `/__ekoa/demo-bridge.js`, injected into every served app via `apps/injected-context.ts`), dashboard player (`web/lib/demo/tour-machine.ts` + `web/components/demos/DemoOverlay.tsx`). Command surface: spotlight/await/annotate/clear/end over origin-pinned postMessage; passive (awaits USER action; does not execute).
- **A data-plane action primitive exists.** `automation/platform-primitives.ts`: every built artifact's `MANIFEST.md` declares capabilities; each capability's recipe is a sequence of deterministic primitives (store.list/get/create…, integration calls) walked by the EkoaActionExecutor (`automation/executor.ts` family, parsed by `automation/manifest-parser.ts`). Server-side, no LLM. UI-level in-page actions do NOT exist.
- **Assistant seam pre-provisioned.** `shared/src/app-assistant.ts` declares `POST /api/app-assistant` (`header-scoped`, minimal message+history→reply) — NOT mounted/implemented; `llm/attribution.ts` has `assistant-chat` kind; `billing/allowance.ts` names served-app assistant chat as a gated surface.
- **Knowledge retrieval ready.** `knowledge/` exports `searchKnowledgeIndex`, `buildGroundingBlock` (cited-or-silent), org-partitioned vault + `_shared` corpus; ingest via service + offline importer.
- **Build pipeline.** `agents/build.ts` (+ `guided-build.ts`) → `apps/build-mechanics.ts` → `scaffoldApp` → agent iterates in sandbox → esbuild → `apps/verify-runner.ts` (playwright-cli drive, `build.verifyBuilds` default on) → served at `/apps/:idOrSlug/` with injected context. Structural instructions live in `api/content/coding-agent/SKILL.md` (~12.7 KB) + scaffold templates.
- **Metering.** `llm/client.ts` single metering point → `billing/tracker.ts` single `token_events` writer; breakdown groups by `agentType`.
- **Store is Mongo** (`api/src/data/mongo.ts`), not Firestore (brief wording is stale).
- **Journey suite** `api/tests/journeys/j*.mjs` (J3 = build probe); suite-ledger census + ratchet binds new specs (`api/tests/SUITE_LEDGER.json`).

## Acceptance criteria (run-level)

1. **Bases wired:** base registry + loader; agent base-selection in the build flow; app base carries action-registry runtime, assistant panel mount, protocol client, design-token link, error boundaries; per-build manifest verification asserts base files were replaced/extended (closes F16/F28 class); structural boilerplate measurably moved out of instructions (token counts recorded vs baseline); J3 passes with a base-built app; verifier fails a deliberately untouched-base build.
2. **Classifier:** scoping gains an artifact-type gate (app vs presentation/report/document); only apps get the operator.
3. **Action registry:** typed manifest of UI commands emitted at build time (navigate/setField/toggle/select/highlight/startTour + per-component actions); dispatch through the app's own state layer; thin client runtime ships in the app base; Cortex assistant receives manifest as tool definitions; destructive actions flagged + client confirmation; every action logs to audit; registry exposed to the tester agent; round-trip gate green on a sample app.
4. **Assistant panel:** side panel in every generated app; three modes (do-it-for-me / show-me / teach-me) inferred from phrasing; first-open message states capabilities (PT-PT); visible driving indicator; any user input immediately pauses assistant driving; domain answers cite knowledge entries; scripted-conversation gate green.
5. **Tours:** declarative tour scripts generated at build time (overview + one per main journey); playback client-side, zero tokens; selectors via registry IDs stable across rebuild; gate green.
6. **Knowledge-during-build:** scoping detects domain-heavy apps, requests uploads, indexes into org knowledge mid-build, narrates it; assistant cites seeded content; gate green.
7. **Metering:** every assistant LLM turn metered + attributed (billing-truth probe extended); tours + registry-only actions free; panel lazy-loads, perf delta within budget.
8. **Security block (9–10, contiguous, last):** capability layer `can()` real implementation; builder→user migration; classifier-as-permission-gate → pre-drafted build request to org-admin; identity/session decided per topology (detect-then-ask stands); edit mode = scoped patch runs with preview/approve/rollback (admins); request-changes queue (users) feeding admin queue; capability-matrix + isolation + server-side destructive-authz assertions; Codex adversarial review over the block in one pass; journey suite fully green. Lands together or not at all.
9. Five-layer QA + suite-ledger + diagram invariant respected on every slice; contract tests for every new/changed endpoint in the same slice; PT-PT copy on all lawyer-facing strings; nothing merges to main (run branch only).

## Non-goals (binding)

## ASSUMPTIONS LEDGER (decisions made on the operator's behalf; chosen vs alternative)

1. **Registry foundation vs extend automations.** CHOSEN: build the UI action registry as the foundation (per-app manifest of UI commands + in-page runtime on the demo-bridge transport pattern), and unify at the MANIFEST level with the existing capability/recipe primitives (UI actions and data-plane capabilities become sections of one per-app operate manifest; automations keep their engine untouched this run — migration is a documented path, not executed). ALTERNATIVE: extend `platform-primitives.ts` into the UI registry (rejected: those primitives are server-side data-plane ops walked by a server executor; UI driving is client-side dispatch — different execution plane, forcing them into one interpreter couples tiers 5 and the served plane). The Phase-2 memo carries the full evidence.
2. **Tour format.** CHOSEN: reuse + extend the existing demo-spec schema (`_schema.json` v1, registry, bridge, tour-machine, DemoOverlay) — add registry-ID selector targets and build-time generation; version bump within the existing validator. ALTERNATIVE: new declarative schema (rejected: a complete, tested, PT-PT player + injected bridge already exists and ~30 authored tours prove the format).
3. **Base set for v1.** CHOSEN: `app` (new, strategic — carries runtime/panel/protocol/tokens/error-boundaries; built from `app-auth-persistent` learnings) + `document` (already authored). `presentation`/`landing`/`app-integration-heavy` stay authored-but-unwired unless the token-tax measurement justifies wiring (wiring is cheap once the loader exists; authoring is not the bottleneck). ALTERNATIVE: wire all five immediately (rejected: verification cost per base; measurement first).
4. **Assistant contract shape.** CHOSEN: extend `shared/src/app-assistant.ts` (breaking-change-free additive evolution where possible: request gains optional mode/context/route/action-results; response gains actions/citations/tour refs; add an SSE-free synchronous v1 consistent with the declared descriptor, streaming only if the panel demands it — decided at implementation with evidence). ALTERNATIVE: new endpoint family (rejected: the served-app plane already reserves this route; contract-first discipline says evolve the descriptor).
5. **Store migration wording.** The brief says "Firestore migration"; the actual store is Mongo (`data/mongo.ts`). CHOSEN: the roles migration (security block) is a Mongo migration. No Firestore work exists anywhere.
6. **`can(capability)` seam location.** CHOSEN: `api/src/auth/capabilities.ts` stub exporting `can(actor, capability): true` (permissive constant, marked `PERMISSIVE-STUB`) + a `shared/` capability name union; Phases 1–8 call it at most from new code paths; real mapping lands in the security block. ALTERNATIVE: scatter TODO role checks (rejected by the brief itself).
7. **Sample app for gates.** CHOSEN: gates that need a generated app build one through the real pipeline (J3-style HTTP probe) against the dev stack, using a seeded org; the fees-app example from the brief becomes the canonical sample (with seeded docs for knowledge gates). ALTERNATIVE: hand-authored fixture app (rejected: gates must prove build-time generation, not hand wiring).
8. **Audit surface.** CHOSEN: assistant/registry actions log through the existing single audit write path (`data/` `logActivity`) with a new activity kind; the "global audit view" requirement is satisfied by the existing activity surface plus filters, not a new subsystem. ALTERNATIVE: new audit store (rejected: one audit path is a standing invariant).
9. **Run branch.** CHOSEN: all work lands on branch `operator-run` off current main (d55bd02 + any local commits); checkpoint commit + tag per slice (`operator/<sliceId>`); merge to main only by the operator after diff review. Stray `web/next-env.d.ts` dev churn is restored, not committed.
10. **Panel implementation plane.** CHOSEN: the assistant panel is part of the served-app plane (injected/mounted from the app base into generated apps), NOT a dashboard (web/) feature; the dashboard is touched only where the brief demands (audit view filters, admin queue in the security block). ALTERNATIVE: dashboard-hosted panel around the preview iframe (rejected: the operator must exist wherever the app is opened, including direct `/apps/:slug` visits).
11. **Model tiers for assistant turns.** OPEN in the brief; CHOSEN default: action-planning + domain Q&A run WORKHORSE; FAST for mode inference/small classification; no EXPERT default (cost). Recorded as tunable config, revisited by the operator with real usage. 
12. **Pricing shape** (per-turn pass-through vs bundled) is a commercial decision — NOT in run scope (brief says so); metering records enough to support either.
