# G12 Codex scope — shared/ contract — triage

Cross-model pass: `codex exec` gpt-5.5, high effort, read-only, over `shared/src/**` (verdict JSON: `codex-checkpoint-shared.json`). 7 findings; **4 fixed + regression-tested, 3 rebutted with rationale.** Claude is the deciding authority — a finding is fixed only where I independently agree it is a real, material contract defect.

## Fixed (4)

- **F1 (HIGH) ErrorEnvelope `details: z.record(z.unknown())`** — the envelope permitted arbitrary internal objects to validate as a legal error body. Fixed: `details` bound to a recursive plain-JSON value (`JsonValue`), so a `Date`/`Buffer`/`bigint`/function/symbol/`undefined` leaf — the accidental-internal-object leak shapes a careless `sendError(..., { err })` would pass — is rejected at the contract boundary. Verified empirically (Date/Buffer/bigint rejected; zod `issues` + `billingUrl` accepted). Honest scope: a schema cannot distinguish a pre-stringified stack STRING from a legitimate string, and an `Error` instance coerces to `{}` (non-enumerable props stripped, leak neutralised); the **primary** control remains runtime FIXED-8 error sanitisation (ch09 §9.3 invariant 2). This is defense-in-depth that also makes the contract test a guard. `errors.ts`.

- **F5 (MED) `AuthUser.passthrough()`** — a RESPONSE shape; passthrough would let a `passwordHash`/reset-token field validate as a legal AuthUser body. Fixed: `.strict()`. Runtime was already safe (`auth/service.ts` `view()` whitelists exactly id/username/role/orgId/active/passwordChangeRequired); strict makes the guard structural — the contract test now rejects a secret-bearing AuthUser. `auth.ts`.

- **F4 (HIGH as filed → real risk LOW) session-capture `session: z.record(z.unknown())` on two RESPONSES** — Codex read the field name "session" as the captured Playwright storageState (cookies). As-built, the field carries capture STATUS metadata (`{status, capturedAt, message}`; the web consumer's `IntegrationSessionStatus.session` shape), the two endpoints are forward-declared with **no server handler yet**, and the web client re-casts — so there is no live leak. But the loose record would let a future handler serialize cookies. Fixed: `SessionCaptureStatus.session` → a bounded `SessionSnapshot` (status enum + capturedAt + message); `ConnectSessionResponse.session` → a bounded status object. Regression test rejects a `{cookies:[...]}`/`{storageState:...}` payload. `integrations.ts`.

- **F6 (MED) DelegatedTask canonicalisation maps non-finite numbers onto `null` bytes** — `budget.egressBytes: z.number()` accepts `Infinity` (zod 3 default), and `stableStringify(Infinity)` → `JSON.stringify(Infinity)` = `"null"`, collapsing two distinct budgets onto identical signing bytes (§18.1 canonicalisation must be injective). Fixed: `egressBytes` → `.finite().nonnegative()` (schema rejects Infinity), and `stableStringify` throws on any non-finite number (belt-and-braces for future signed fields). Regression test: Infinity rejected at schema + canonicaliser throws + distinct finite budgets → distinct bytes. `ekoa-local.ts`.

## Rebutted (3) — real behavior already enforces the invariant

- **F2 (HIGH) automation `RunCreateRequest.inputs: z.record(z.unknown())` "smuggles reserved credential material."** The `credentials` key is the **intentional** in-memory-only channel for a captured browser `storageState` (ch05 §5.6.7, v2 invariant I2). Runtime: `engine.ts` `scrubCredentials(inputs)` strips it from the persisted run record; template-vars redacts it from any substitution; the run executes in the **creator's own** security context (A4), so there is no cross-actor escalation. Rejecting `credentials` (Codex's recommendation) would break the feature. Per-automation input-schema validation is a runtime service-layer concern over a dynamic, user-authored schema — not statically expressible in the shared contract. No change owed.

- **F3 (MED) integration `CreateConfigRequest.configValues: z.record(z.unknown())` "arbitrary key/value bags."** Provider config values are provider-specific by design; they are **encrypted at rest** and **never returned** (`configSummary` is metadata-only: id/integrationKey/name/enabled/needsReauth/ownerUserId — G4 established creds-never-returned). Binding to a per-definition credential schema is impossible statically (integration definitions are dynamic/user-created). The open bag is intentional and its risk is closed at persistence + response. No change owed.

- **F7 (MED) bridge provider `credential: z.string()` "unbranded — a session token could validate as a bridge credential."** Two-class token separation (ch09 §9.2) is a runtime **cryptographic** property: the platform verifier positively rejects bridge tokens (aud/connectionId), the bridge verifier rejects session tokens, and the provider endpoint's credential→pairing→org guard (G8A) binds a credential to its socket+org. A zod `.brand()` is a compile-time TS phantom with **no runtime effect**, and the wire legitimately carries an opaque token string; the contract does not (and cannot) encode the crypto class. Enforcement is at verification, already tested (47 bridge + fake-daemon tests, F1/F2 socket-binding fixes). No change owed.
