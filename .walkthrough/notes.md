# Walkthrough notes — ekoa

## Flows that matter
- The build arc is THE product story: chat request -> [[EKOA_BUILD]] handoff -> building -> finished artifact -> preview renders -> follow-up change takes effect.
- Knowledge chat: a real legal question answered with corpus citations (262k-doc `_shared` corpus) is the differentiator; always show the citation.
- Automations plan-from-goal: goal in, real plan steps out.
- Brand research: run it, then PROVE persistence (org branding after reload / GET /org evidence panel).
- Integration-builder: chat handoff or panel -> generated package -> saved -> visible on /integrations.

## Captions
- PT-PT captions (the product UI is Portuguese; walkthroughs double as product material).
- Plain language; never internal codenames (no "chokepoint", "seam", "FIXED-n").

## Always show
- The verified result highlighted (the artifact rendering, the citation, the persisted branding), not just the submit.
- Evidence panels for persistence proofs (GET /org, GET /artifacts) with the proving line highlighted.

## Avoid on camera
- Login typing (use .walkthrough/auth.json storageState).
- Terminals, test runners (banned by the skill anyway).
- /users and /usage token figures are fine (demo instance), but never credentials/secrets screens.

## Selector gotchas (learned 2026-07-11)
- getByRole name matching is SUBSTRING: `button:Marca` also hits "Pesquisar Marca". Use `role:tab:X` for tabs, exact-CSS (`span:text-is("...")`) for step labels.
- `text:Passo 1` double-matches (span + wrapper div render the same text) and substring-hits "Passo 10+".
- Chat run-start signal: `[title="Cancelar build"]` (the queue button only exists when text is typed mid-run). Run-end: `[title="Enviar mensagem"]` reappears.
- /chat auto-resumes the latest session: DELETE walkthrough sessions via API before recording chat flows, else the empty-state composer never shows.
- Automations plan-from-goal AUTO-STARTS a rehearsal run on the editor: two Cancelar buttons mid-run (`button.ml-auto.border-red-300` is the activity-bar one); engine caches rehearsal step results across repeated runs of the same plan.
- Integration builder is a TWO-TURN handshake in chat: turn 1 the agent OFFERS ("Podemos construir uma integração com o {Serviço}. Queres que comece já?"), turn 2 the user confirms ("Sim, começa já...") → the [[EKOA_INTEGRATION_BUILD]] marker fires → the side panel mounts and generates. A single-turn "cria uma integração..." gets misclassified as an app build. Signals: offer `text:Queres que comece`; panel `text:Integration Builder`; ready `text:Integration package ready to save`; saved `text:está pronta`; card on /integrations `text:Countries`.
- Delete the created automation/session between takes (duplicate names → ambiguous asserts).
- Chat page has a persistent SSE: a `continue + speed:8` timelapse segment on /chat CRASHES ("produced no parseable result"). Use a `waitBefore` CUT (wait off-camera for the signal, then record statically) instead of timelapse on chat.
- AI-built integrations land under `/integrations?tab=plataforma` (NOT `?tab=minhas`, which is empty). The card `h3` is the displayName; the per-card "Mostrar mais" toggle is AMBIGUOUS (one per card) — scope to the card or skip expansion and prove actions via an evidence panel (GET /api/v1/integrations filtered).
- The chat→integration confirmation deterministically also spawns a failed app build (ledgered `integration-handoff-spurious-build`): keep the confirm+generate off-camera (cut) and feature the saved card + evidence to keep the video clean.
- BUILD walkthrough: recording a live 5-15 min build via `continue`+`waitBefore` on the chat page CRASHES (WS-heavy). Instead: (1) fresh chat segment shows the request + the working indicator — assert on `[title="Cancelar build"]` (stable while running), NOT the transient `text:Construindo app`; and keep the request SHORT (a long `fill` on the auto-resizing landing composer times out `pressSequentially`). (2) Title-card CUT. (3) A FRESH browser segment on `http://localhost:4111/apps/<artifactId>/` (the served app — permissive CORS, no dashboard SSE) shows the real built app rendering + interact + assert (use `text:IVA (23%)`, NOT `text:Total` which also matches "Subtotal"). (4) evidence panel: `curl /apps/<id>/` proves it is served HTML. Pre-build the artifact off-camera and hardcode its `/apps/<id>/` URL.
- 2026-07-11: five verified walkthroughs recorded (automations, knowledge, brand, integration, build). Gallery http://100.108.210.116:8099. Superseded/flagged takes pruned so only the 5 verified finals remain.

## Operator-suite lessons (2026-07-14, 3 verified walkthroughs: panel / edit-mode / pedidos)
- Fresh app-base apps are PRIVATE: plain /apps/<id>/ 410s until `PATCH /artifacts/:id {shareable:true}`; `visibility:'org'` additionally gates change-request filing + org-admin edit (both needed for the operator flows).
- `POST /users` WITHOUT orgId puts the user in a NEW org - pass the admin's orgId explicitly (request-changes-journey pattern) or the pedido file 404s on cross-org isolation. Re-mint the auth state after recreating a user (the JWT carries sub+orgId).
- Panel auth on :4111: craft storageState with `ekoa_token` on BOTH origins (:3000 + :4111) and strip `ekoa_orchestration` (stale session pointers 404 on camera). `playwright-cli state-save` resolves relative to the CALLER cwd - use absolute paths.
- Recorder gotchas: `goto` paths inside continue segments MUST be absolute URLs (relative never navigates, beats fail downstream); a goto INSIDE a recorded segment poisons the NEXT continue segment's reattach ("produced no parseable result") - make the following segment FRESH (re-open with authState) instead; `text:X` waitFors can match a CLOSED select's option text (invisible - times out) - scope waits to `[data-testid=...] tr:has-text(...)`.
- Edit-mode filming: the served page never auto-reloads after a patch run - approve/revert from the sha diff, THEN reload to show the effect; the post-RESTORE dist rebuild is LAZY (~60-90s) - bridge with a speed-8 continue segment of captioned reload cycles; generated task apps HIDE bulk buttons on an empty list - seed a row before asserting them.
- Pedidos queue keeps take residue - dismiss stale open pedidos (POST /change-requests/:id/dismiss) between takes; the chat-refusal pedido carries the AGENT-drafted build description, not the user's literal message (by design).
- This stack's patch runs took ~2-8 min (faster than the 12-17 min worst case); build the request text to name button labels in quotes so asserts are deterministic.
- Stack ops: the model credential is per-boot (`provision-credential.mjs` with the token from ~/.config/ekoa/claude-credentials.json accessToken); panel-runtime.js and action-runtime-client.js are cached in memory at first serve/boot - asset fixes go live only on restart (restart wipes the DB: rebuild fixtures + re-provision + re-mint auth).

## Voice (Part C) EVIDENCE walkthrough lessons (2026-07-18, mega-run 20260717-190134-9d4c1cbf)
- Mocking getUserMedia + a WS marker-frame relay (the voice-proof.spec.ts / voice-modality.spec.ts pattern) is injectable into the recorder via a NEW opt-in `initScript` field on a browser segment (raw JS string, applied through `page.addInitScript` before the segment's first navigation) - added to the shared skill's `browser.mjs`/schema, not app-specific. Two companion opt-in beat actions were added alongside it: `evaluate` (runs a JS expression string via `page.evaluate`) and `mousePress` (a real `page.mouse.down/wait/up` hold gesture, for a press-and-hold affordance a plain `click` can't express).
- **`page.evaluate(scriptString)` does NOT auto-invoke an arrow-function-source string** - it evaluates the string as a bare expression, which just constructs-and-discards the function. The `evaluate` action now wraps every script as an IIFE call (`(${script})()`) before evaluating, so storyboards just supply a plain `"() => ..."` / `"async () => { ... }"` source and it actually runs. (Diagnosed via a marker global that stayed `null` until wrapped.)
- **Re-minted `.walkthrough/auth.json` can carry a stale `ekoa_orchestration` localStorage session pointer** from whatever session existed at `state-save` time (even one deleted seconds later via API) - strip that key after minting, same as the operator-run notes already flagged for panel auth. Left in place it isn't fatal but adds an extra (sometimes slow) session-list detour before the URL's own session resolves.
- **A fresh `/chat/<sessionId>` deep-link can render the generic blank "O que vamos construir" landing state instead of the requested session** on the very first paint, and - once it happens - it does NOT self-correct even after 70+ seconds; a `page.reload()` immediately after the initial navigation reliably fixes it. This looks like a client-side hydration race (an initial default-session fetch competing with the URL-driven one) that is sensitive to some timing factor not yet isolated (not reliably reproduced in smaller non-recorder repros, but hit consistently through the full recorder pipeline on brand-new session ids). Added as an opt-in `settleReload: true` browser-segment field (one extra reload before any beat runs) rather than changed default behavior. **If this resurfaces, it may be worth filing as a real app-level finding** (docs/findings.md) rather than just papering over it here - the recorder workaround hides it but the underlying race is in the product, not the tooling.
- The stub TTS's audible "Parar" window can be shorter than the recorder's fixed frame-sampling offset (caption-time + up to 1.4s) for a short reply - the functional `assert` (which polls, so it reliably catches the toggle) can pass while the extracted verification FRAME lands just after it reverted to "Ouvir". Reproduced identically across two independent full recordings with different live-model replies, so it's a structural sampling-vs-playback-duration mismatch, not a fluke - record it `evidence-degraded` rather than chasing a re-record.
- The chat page's composer is CSS-fixed to the viewport bottom, so a highlight on `voice-mic-button` sits inside the recorder's 96px caption-bar zone regardless of viewport height (confirmed identical at both 800px and 900px) - a taller `video.height` does NOT create clearance for a bottom-pinned element. Highlighting a higher, non-fixed ancestor (e.g. `voice-bar`) avoids the clamp; highlighting the mic button itself on a page with no open voice-bar does not.
- `armMarker`/`fireVadSpeechStart` scripts must be written as bare function-expression sources for the `evaluate` action (see the IIFE-wrap point above); `fireVadSpeechStart` should poll for `window.__voiceTestVad` to exist (up to ~10s) before calling it - the driver only wires that global up after `capture.start()` + `stt.open()` resolve, so firing immediately after the UI shows "listening" can race ahead of it (same race the spec's own helper documents).

## Part E (portal connectors) EVIDENCE walkthrough lessons (2026-07-18, mega-run 20260717-190134-9d4c1cbf)
- A backend connector feature whose only external dependency is a REAL third-party portal (here: government certidão/Citius insolvência portals) cannot be driven live through the served-app UI in a walkthrough without either real network egress + a real access code (not available), or wiring a fixture-backed fetchImpl into the long-running boot-b/dev stack (nontrivial - boot-b's ephemeral mongo URI isn't exposed for an external seed script either). When a slice's own committed gate test (e.g. `api/tests/legal/e5-portal-gate.test.ts`) already exercises the real router + real CollectionsEngine over mongodb-memory-server with an injected `fetchImpl` reading committed HTML fixtures, the honest and practical evidence source is a THROWAWAY vitest spec that mirrors the gate's exact setup (same imports/fixtures/assertions) but additionally `writeFileSync`s the real request/response bodies + the real persisted documentos/eventos rows to a JSON file - run once OFF-camera (`npx vitest run` from the temp spec, never shown on screen), then delete the temp spec. This is materially different from "running the test as evidence": the captured artifact is real HTTP responses and real stored rows, not a pass/fail.
- The evidence panel's `buildRows` auto-windows long file/command content to a budget centered on the `highlight.match` line, so a big raw JSON dump mostly works - but when the criterion needs SEVERAL fields visible at once that are far apart in a large dump (e.g. `origem` near the top, `parsed.nif` mid-file), it's cleaner to hand-curate a SMALL derived JSON (a few captured real fields only, no fabricated values) into its own file per beat, sized to comfortably fit the panel's ~18-19 row budget at 1280x800. This is the same "curl | jq" idiom the schema doc's own evidence example uses, just pre-baked instead of run live.
- A pure EVIDENCE-mode walkthrough can be ALL `evidence` segments with no `browser`/`title` segments at all - `record.mjs` handles each segment type independently; no browser session is required when the proof is API-response/stored-row panels only.
- `evidence` segments support `file` + `cwd` (both from `references/storyboard-schema.md`'s own example) pointing anywhere on disk, including outside the repo (e.g. the session scratchpad) - useful for off-camera-captured JSON that has no reason to live in the target repo.
## OS-mode walkthrough lessons (2026-07-18, 1 verified walkthrough: os-mode)
- The recorder's `drag` op needs the TARGET to report an `over` class (dnd-style closed loop). Pointer-capture drag surfaces (window titlebars, resize handles, dividers) never hover the target - the OS window layer now mirrors its live drop candidate as an `over` class on `os-snap-left/right`, so window snap drags confirm closed-loop. For non-drop drags (panel resize, divider), point `overClass` at a class the target always has (e.g. `font-display` on the page title, `absolute` on the snap strip).
- NEVER pick artifact cards by sort position for rename/delete: fork copies tie on updatedAt and the ORIGINAL can sort first (a positional pick renamed + deleted the seeded Portfólio Agência). Scope to `div.group:has(h3:has-text("(cópia)"))` and clean copies via API between takes.
- The OS dock sits under the caption HUD in every captioned frame - dock beats read best through their ACTION and PAYOFF (window opens from the pin); the dock itself is visible in caption-gap moments only. Expect dock-focused `expectedScreen`s to be about the clamped highlight, not the strip.
- Take-2's recording caught a REAL bug e2e width-asserts missed: imperative left/top fighting React style reconciliation stranded snapped windows off-screen (fixed: live drag rides `transform`). Walkthrough gates earn their keep.
- The narrow-viewport scenario cannot be filmed (recorder has no per-segment viewport); it stays covered by the committed os-mode spec S5.
- PT-PT copy check applies to CAPTIONS too: no travessões ("—") - two takes were burned on em dashes in a caption + the outro subtitle.
