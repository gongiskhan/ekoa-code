# 11. Glossary: old vocabulary to new

This chapter exists to prevent regression of the frame. The old Cortex described itself in a private vocabulary (its "content-driven" doctrine), and any future agent session that reads old Cortex code, its docs, or its commit history will meet that vocabulary and be tempted to rebuild what the words imply. This is the ONLY chapter in the spec where the old terms appear, apart from SPEC.md's FIXED register, which preserves the founder's original wording verbatim; everywhere else the spec uses plain conventional vocabulary (CONV-6, SPEC.md conventions register). For each old term the table gives: what it actually was, its replacement, where the replacement is specified, and the concrete failure mode if the old concept is resurrected. When an old-code comment, reference doc, or migration script mentions one of these words, translate it with this table and proceed in the new vocabulary.

## 11.1 The mapping table

| Old term | What it was | New term | Where specified | Failure mode if resurrected |
|---|---|---|---|---|
| intent, dispatch, `POST /api/v1/action` | One generic endpoint taking `{app, intent, params}` envelopes, routed by string name to a handler (reference/operations-inventory.md section 0.1) | Resource-oriented REST endpoints, one route per operation | ch03 section 3.1 (FIXED-2) | A single do-anything endpoint destroys per-route typing, per-route auth, and contract tests; FIXED-2 bans it outright |
| app/intent envelope, recipe envelope | Request wrapper `{app, intent, params}` and the sometimes-present response wrapper `{success, recipe_id, data}` the client had to detect and unwrap (reference/operations-inventory.md section 0.2) | Typed routes: one zod-validated request and response shape per endpoint, defined in `shared/` | ch03 section 3.1 (FIXED-1) | Envelopes hide the real payload from the schema layer; the dual-shape unwrap bug class (landmine 1) returns |
| global `GET /api/v1/events`, SSE event bus as protocol | One global per-user stream carrying every event type; the app's de facto transport | Four scoped SSE streams, one per genuine stream (chat run, build job, automation run, notifications); everything else request-response | ch03 section 3.6 (FIXED-2) | A grow-only global stream becomes an untyped side channel; every new feature quietly adds events instead of designing an endpoint |
| `trace_id` (client-minted) | Client-generated correlation id used to address and cancel runs | Server-minted `runId` / `jobId` returned by the creating POST | ch03 section 3.4 | Client-minted ids break owner-scoped cancel and duplicate-submit protection |
| chat `mode` enum (`force_local`, `force_external`, `force_orchestrated`) | Client-sent routing directive telling the backend which execution path to take | Server-side tier classification inside the LLM chokepoint; the client sends content, never routing | ch03 Appendix A; ch06 (FIXED-3) | Routing decisions leak into the client and every client must reimplement them; FIXED-10 client-agnosticism breaks |
| recipe, recipe app, `recipes.json` | JSON files declaring deterministic CRUD operations, loaded at boot and interpreted at runtime by a generic engine (with a runtime compile endpoint) | Split by nature: user-app data CRUD becomes the collections engine; platform domain operations become typed REST routes backed by plain TypeScript | ch04 section 4.2 (FIXED-5); ch03 section 3.8 (FIXED-4) | Reintroduces runtime interpretation of declarative artifacts and the compile-at-runtime machinery; FIXED-4 requires code authored at design time |
| instruction (runtime-interpreted markdown workflow) | Markdown workflow files a model read and executed at runtime as platform logic | Typed pipelines: multi-step workflows written as plain TypeScript in domain modules | ch02 section 2.6 (FIXED-4) | Puts an LLM inside a runtime platform path (routing, lifecycle, CRUD); violates FIXED-3 and FIXED-4 |
| executor | The runtime layer that located and ran content (recipes, instructions) on demand | Modules: ordinary imported TypeScript per the module map | ch02 section 2.6 | Implies a generic execution layer over content; there is none - code is called, not "executed by" anything |
| handler + domainMap | Thin TypeScript dispatchers keyed by `ekoa.{domain}` app id, registered in one central name-to-function map | Express routers, one file per domain resource; routes declared statically | ch02 `routes/` entry; ch03 section 3.8 | A central string-keyed map is the generic dispatch surface by another name; route census (ch03 acceptance 3) becomes unauditable |
| governed tool, tool registry (`registerGovernedTool`) | Privileged operations (crypto, JWT, files, VCS) wrapped in a runtime registration/audit shell | Plain modules with review gates: ordinary functions in their domain module; enforcement moves to lint rules, the single audit write path, and code review | ch02 section 2.9; ch09 (FIXED-8) | A runtime permission registry adds machinery without adding enforcement; the real invariants (chokepoint lint, boundary zones, audit path) already cover it |
| orchestrator, phases, `phase_changed` | The orchestration entry point wrapped around agent machinery, emitting phase events (dead on the wire today, reference/operations-inventory.md C5) | The chat run pipeline and its typed run states and events; phase information folds into job status events (RESOLVED (P-11), ch05) | ch05; ch03 sections 3.6.1, 3.8.7 | A named "orchestrator" invites a generic coordination layer; runs are jobs with states, streamed on scoped endpoints |
| adapter (`adapters/external.ts`) | One 1600+ line module mediating every agent call: SDK invocation, prompt assembly, streaming, metering, marker parsing | Split: `llm/` is the Anthropic chokepoint (client, attribution, metering); `agents/` owns job lifecycle, context assembly, and typed streaming | ch06 (FIXED-3); ch05 | A monolithic adapter re-fuses metering with orchestration; the no-Anthropic-outside-`llm/` lint (ch02 section 2.9) becomes unenforceable |
| content-driven | The doctrine that business logic lives in markdown/JSON content which the platform loads and executes | Agent-context content: files whose ONLY consumer is agent prompt assembly via the `content/` loader | ch08 (FIXED-6) | The moment any platform code path reads a content file to decide runtime behavior, FIXED-4 and FIXED-6 are both broken |
| skill (as backend machinery), skills manifest, skill loader | SKILL.md files auto-discovered at boot and treated as platform capability units, with a manifest and loader machinery | Agent-context content consumed by the `content/` loader (per-user composition directories plus content-addressed cache); the term "skill file" survives only in ch08 for context assembly | ch08 (FIXED-6, FIXED-7) | Skills-as-capability-registry recreates the framework; content shapes agent prompts and can never define routes, schemas, or data transformations |
| JsonStore | Whole-file JSON persistence at `~/.ekoa/data/` with atomic rename, one file per domain | Data layer stores: Firestore collections behind `api/src/data/` (RESOLVED (P-05)), with named filesystem/SQLite exceptions | ch04 sections 4.3, 4.4 | New whole-file JSON stores reintroduce the write-amplification hotspot (token ledger) and a second persistence idiom |
| `ekoa.templates` domain, artifact instance | The domain that managed templates and "artifact instance" records under one handler | The artifacts resource: `/api/v1/artifacts`, resource noun "artifact"; "template" survives only as the optional starting scaffold (`templateId` on job create) | ch03 section 3.8.9 | "Instance" vocabulary drags the template/instance split back into the API and doubles the resource model for no gain |
| `memory/anonymizer.ts`, `anonymizeContent`/`anonymizeMemory` (the old "anonymizer") | A dead 53-line regex PII scrubber in the memory subsystem (URLs/paths/emails/IPs/ports/tokens/UUIDs to placeholders): zero production call sites, referenced only in comments, and the commit guard explicitly declined to reuse it because it false-positives on hashes/UUIDs/base64 (reference/carryover-audit.md `memory/anonymizer.ts` row; reference/invisible-behaviors.md Conflicts #15) | The anonymisation layer: the egress tokenization pipeline built inside `llm/` (detection, per-session in-memory vault, de-tokenization including tool_use blocks, hash-chained metadata-only audit) - greenfield, NOT descended from this module | ch17; ch02 section 2.6 (`llm/`); FIXED-8 (as amended), FIXED-13 | Carrying the old regex stub as, or into, the anonymisation layer gives false PII-safety confidence: it detects the wrong things, sat on no egress path, and has none of the vault/tokenization/audit mechanism. It is a drop candidate, not an ancestor; the new pipeline is built fresh (chapter 17) |
| `teams` (teams domain: endpoints, pages, store) | The teams grouping primitive - its endpoints, pages, and store - a second organisational axis standing beside the firm | Deleted end to end (Amendment 2); departments, if ever demanded, return as additive groups inside an org | ch03 Appendix A; ch04 §4.3.1 DROP row | Resurrecting a parallel grouping primitive beside the org model re-creates two ownership dimensions and breaks the org-scoped sharing model |
| `tenant` / `company` | The old single-company singleton and the informal "tenant" | **org** (schema/API name; PT-PT display label "Escritório"); the org record carries identity, branding, and org settings | ch04 (orgs collection); ch03 §3.8.4 | Keeping a company singleton or an undefined "tenant" resurrects the single-firm assumption and unscopes branding, knowledge, and deny-lists |
| `license`, license tiers, tier feature map, license server | The licensing plane: a separate license server, per-install license records, tiers, and a tier-to-feature map | The activation model: `active` boolean + the billing allowance gate; no tiers, no license service | ch09 §9.7.1; ch03 §3.3 | A license plane invites boot-time global gates and dead tier machinery; access control belongs on the per-request activation + billing gates |
| Supabase control plane (`standalone_credentials`, `companies`, `installations`) | The external Supabase-hosted control plane holding the three live tables | Firestore: the `credentials` singleton (encrypted), the org model, and the activation model | ch04 §4.5; ch10 migration rows | Re-adopting an external control plane splits custody across systems and re-creates the coexistence mechanics chapter 10 retired |

## 11.2 Fates of the three bundled recipe apps

The old repo shipped exactly three recipe apps (reference/operations-inventory.md C1). Each has an explicit fate; none carries as a "domain app":

| Old recipe app | What it was | Fate | Where specified |
|---|---|---|---|
| `ekoa.company` | Company/branding singleton read through the recipe interpreter, with its data file trapped inside the recipe app's data path (reference/data-inventory.md section 5.1) | Lives on as the org REST resource (`GET`/`PATCH /api/v1/org`, `PUT /branding`) over the `orgs` collection (Amendment 2; the old `company` singleton is superseded by the org record, PT-PT label "Escritório") | ch03 section 3.8.4; ch04 section 4.3.1 |
| `ekoa.projects` | Projects CRUD with zero consumers outside its own persistence module (reference/data-inventory.md section 5.1) | DROPPED. Not migrated, no replacement resource | ch03 Appendix A (C2); ch04 section 4.3.1 (DROP row) |
| `ekoa.deployments` | Bundled recipe app whose store (`deployments.json`) has zero code references at HEAD (reference/data-inventory.md conflict C2) | DROPPED. Not migrated, no replacement resource. Artifact start/stop lives at `/api/v1/company-space` (ch03 section 3.8.12) | ch03 section 3.8.12 covers the surviving behavior |

The runtime recipe-compile endpoint (`POST /api/apps/:appId/compile`) is dropped with the layer it served (ch03 Appendix A; FIXED-4). Do not scaffold a projects or deployments domain "for completeness": both were dead code, and resurrecting either fails ch03 acceptance criterion 3 (route census equals the ch03 map).

## 11.3 Usage rule for the new repository

- The old terms in this chapter must not appear in the new repository (`ekoa-code`) - not in code, identifiers, route paths, docs, or the repo `CLAUDE.md` - except when quoting history or naming an old-Cortex artifact during migration work (ch10). This rule lands in the new repo `CLAUDE.md` from day one, alongside the boundary rules and the diagram invariant that ch02 section 2.2 already places there (FIXED-1, FIXED-12).
- When reading reference material or old-Cortex code during the implementation run, translate on sight: the reference docs in `spec/reference/` intentionally use the old vocabulary because they describe the old system; every spec chapter, and all new code, uses only the right-hand column of section 11.1.
- If a concept appears that this table does not cover, it is not license to adopt the old frame: name it conventionally, and if its placement is genuinely ambiguous, it belongs in the Open questions register (ch16), not in resurrected machinery.
