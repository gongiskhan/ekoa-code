# 01. System overview

One page. If this chapter does not fit in your head, the spec has failed its first acceptance criterion.

## What Cortex is

Cortex is the backend of Ekoa: a platform where a law firm (today; other verticals later) chats with an AI assistant, has apps built for it by a coding agent, runs those apps as served web artifacts, automates browser and integration work, and gets billed for the model tokens its work consumes. The rebuild replaces the current backend (the old architecture, mapped term by term in chapter 11) with **one conventional Node.js + TypeScript REST service** (FIXED-1, FIXED-2). Same product. Boring architecture.

## The one paragraph that explains everything

A single Express service exposes a resource-oriented JSON API under `/api/v1` (auth, users, sessions, chat runs, jobs, artifacts, integrations, automations, memory, knowledge, billing, and the served-app data plane). Every platform operation - routing, CRUD, lifecycle, orchestration - is deterministic TypeScript: no model call ever sits in a platform path (FIXED-3, diagram 03). The model works in exactly one place: agent runs (chat turns, app builds, automation vision, assistants) executed through the Claude Agent SDK, and every one of those calls passes through one egress module, `api/src/llm/` - simultaneously the billing attribution point (tagged at the call site as `user_work`, `classifier`, or `platform`, which is why billing attribution is finally computable), the anonymisation pipeline that tokenizes detected sensitive spans before any payload reaches Anthropic and de-tokenizes on return, and the provider-routing config point (region and zero-retention posture as configuration, never hardcoded): one egress module, three concerns (FIXED-3, FIXED-13, diagrams 04, 06, 10). User apps store data through a collections engine: the app's manifest declares its collections, one generic deterministic API serves all of them over Firestore (FIXED-5, diagram 05). Agent-context content files still exist (chapter 08), but only as *content* a small loader assembles into agent contexts - never interpreted at runtime, never a framework (FIXED-4, FIXED-6, diagram 07). Local firm files never upload: the hosted agent delegates a task to the paired ekoa-local daemon (`delegate_to_local`), which works the files in place and returns only derived output - summaries, citations, patches - so raw content never enters hosted context or persistence (chapter 18). Two trust boundaries govern the data: file excerpts cross the user-machine-to-Cortex boundary in cleartext, transiently, only inside completion requests; nothing sensitive crosses the Cortex-to-Anthropic boundary except as tokens, and the token-to-value vault never leaves Cortex (chapter 17, diagram 10). The existing Next.js frontend migrates into `web/` with a typed REST client generated from the `shared/` zod contract; the UI is the functional contract, the wire protocol is not (FIXED-9). The platform is multi-org: every user belongs to an org (schema/API name `org`, PT-PT label "Escritório"), branding, the knowledge base, integrations, and anonymisation deny-lists are all org-scoped, a `visibility: private | org` field governs whether a memory or artifact stays owner-only or is shared across the org, and access to the whole surface is gated by two independent facts - an admin-controlled `active` flag and the billing allowance, the activation model that replaces licensing (chapters 04, 09).

## The five load-bearing structures

1. **Repo** (`ekoa-code`, a sibling folder of `ekoa-dev`): `api/` (the service) + `web/` (migrated frontend) + `shared/` (zod API contract, the only thing both may import) + `spec/` (this document set). Boundaries lint-enforced. (Chapter 02, diagram 02.)
2. **Protocol**: request-response REST + four scoped SSE streams (chat run, build job, automation run, user notifications). No generic command endpoint, no event-bus-as-protocol, and (FIXED-2, as amended) no WebSockets between frontend and Cortex as API transport - with one scoped exception, the live browser canvas media channel (frames down, input events up, short-TTL token, never JSON API payloads; Q-01 resolved as this carve-out). The daemon-to-Cortex bridge WebSocket is a separate transport, outside the frontend rule entirely (chapter 18). (Chapter 03.)
3. **Data**: Firestore (Mongo-compat driver) for app data via the collections engine and, RESOLVED (P-05), for platform domains; filesystem/SQLite stay where they objectively fit better (knowledge vault + lexical index, sandboxes + git, event queue, blobs); model-provider credentials are centrally managed in a Firestore collection, encrypted, with two auth modes as configuration (subscription OAuth / API key; FIXED-8 as amended), and Supabase is retired (P-08 re-resolved; chapter 10 owns the migration and retirement). Org plus user tenancy bound on every query; single multi-tenant process (FIXED-8). (Chapter 04, diagram 05.)
4. **Agent execution**: jobs are created by POST, gated by billing, run in-process through the Agent SDK, stream over SSE, and end in exactly one terminal state; cancel is a server-side abort. All model access via `llm/`; metering rides the same path, so a user's bill is the sum of their weighted `user_work` tokens. (Chapters 05-06, diagrams 04 and 06.)
5. **App pipeline**: the coding agent writes JSX into a per-user sandbox, esbuild bundles it, Cortex serves it statically at `/apps/:slug/` with the `window.__ekoa` handle injected - preserved byte-compatibly, because 37 of the 55 surviving e2e specs (the whole legal vertical) test through it. (Chapter 07.)

## What is deliberately gone

The runtime content-interpretation layer, the model-in-the-loop request routing, the runtime permission-registry shell, the generic `/api/v1/action` command endpoint and its request envelope, the firehose SSE bus, and all six platform-attributed model calls (each has an explicit fate in chapter 06). The glossary (chapter 11) names each of these in the old vocabulary and exists so no future agent session resurrects them. Amendment 2 retires more: the teams domain (deleted end to end; a glossary row records it), the Supabase control plane, license tiers and the tier feature map (replaced by the activation model, chapter 09), the multi-subscription credential rollover machinery, and the filesystem app-data backend with its `getAppDataBackend()` switch.
