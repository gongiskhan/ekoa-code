# 09. Security invariants

This chapter is the security contract of the rebuilt service: the eleven invariants carried from today's Cortex (FIXED-8), what each one is, exactly where it is enforced in the new module map (chapter 02), by which mechanism (boot gate, middleware, lint rule, single write path, test gate, review gate), and what changes versus today. One carried invariant is amended by this run rather than merely ported: the egress anonymisation invariant (invariant 2) is no longer a seam but a built layer, whose full specification lives in chapter 17 and whose invariant and enforcement homes this chapter states. On top of the carried set, section 9.8 folds in the security baseline (FIXED-14) - the OWASP-LLM/agentic and web/API controls the port left implicit, now each stated with an enforcement home - and section 9.9 records the certification-phase controls deliberately deferred. The chapter also fixes the auth model summary, the webhook ingress security pipeline, the JWT-in-URL surfaces (Q-05, resolved), the workspace Graph-proxy gate (Q-10, resolved), and replaces the former no-revocation tradeoff with the server-side revocation list decided at P-03. Every load-bearing claim cites the reference docs; conflicts between old documentation and old code are recorded, never silently resolved. Two new diagrams in this amendment render this chapter's privacy and delegation boundaries directly - the two trust boundaries in `spec/diagrams/10-privacy-boundaries` (diagram 10) and the S1-S6 delegation bindings in `spec/diagrams/11-delegation-security` (diagram 11); the request-path and module enforcement points remain rendered across `spec/diagrams/02-module-map` (module homes and boundaries) and `spec/diagrams/03-request-crud` (the request-path middleware chain: auth, validation, error envelope). Under FIXED-12, any change to an enforcement point in this chapter updates whichever of those four diagrams renders it, in the same unit of work.

## 9.1 Enforcement vocabulary

Every invariant below names one or more of these mechanisms. An invariant with no named mechanism is not accepted into this chapter.

| Mechanism | Meaning | Checkable by |
|---|---|---|
| Boot gate | The process refuses to start when the condition fails | integration test that boots with the bad config and asserts exit |
| Middleware | An Express middleware applied before route logic | route census + contract tests (chapter 13) |
| Lint rule | ESLint rule in the repo config, CI-failing | CI run |
| Grep gate | A CI step that greps the tree for a forbidden pattern | CI run |
| Single write path | Exactly one exported function may write the resource; direct store access is lint/grep-banned outside its module | grep gate + unit test |
| Data-layer helper | The only query-construction point for a store applies the scope; no raw driver access outside `data/` | lint rule (routes never import `data/`, chapter 02 section 2.9) + parity tests |
| Test gate | A named automated test that fails when the invariant is violated | chapter 13 inventory |
| Review gate | A standing rule in the new repo `CLAUDE.md` that review must reject violations | human/agent review; backstop only, never the sole mechanism except where stated |

## 9.2 Auth model summary

The full endpoint-by-endpoint auth classes live in chapter 03 section 3.2; this section fixes the token architecture that backs them.

**Token classes.** Three mutually non-interchangeable credential planes, plus one machine key:

| Class | Carrier | TTL | Verifier behavior |
|---|---|---|---|
| Platform session JWT | `Authorization: Bearer` header (query `?token=` only on the surfaces in 9.5) | 24 h, or 30 d with `rememberMe` (carried; reference/invisible-behaviors.md section 1.3) | HS256 against the JWT secret; **positively rejects bridge tokens** (any `connectionId` claim or `aud: ekoa-bridge` fails verification) (reference/invisible-behaviors.md section 1.1) |
| Bridge token (ekoa-local daemon) | Bearer header on the WS upgrade (query fallback slated for removal, chapter 03 section 3.10) | 600 s default | audience-checked `aud: ekoa-bridge`; the `connectionId` claim must equal the URL path segment; the bridge verifier rejects session tokens and vice versa - two classes, one secret, never interchangeable (reference/invisible-behaviors.md section 1.5) |
| App-SSO session (end users of served apps) | Per-app HttpOnly cookie, high-entropy opaque id, never a JWT | 8 h (pending-auth state 10 min) | server-side record lookup scoped to ONE canonical appId; wrong-app/expired/missing all return the same 401 (reference/invisible-behaviors.md section 1.8) |
| Gateway key | `X-API-Key` on the ekoa-local LLM gateway | static | constant-compare against the configured gateway key; platform JWT also accepted (reference/invisible-behaviors.md section 2.8) |

**Single mint point.** All platform JWTs are minted by one function in `auth/` carrying `{sub, role, scope, orgId, username}` - `orgId` replaces the former `companyId`, since Amendment 2 defines the tenant as the org (Part 4; founder, 2026-07-06 (amendment 2, consolidated ledger, docs/ekoa-code-spec-amendment-2-consolidated-ledger.md)). The role vocabulary is exactly three values - `super-admin` (platform), `org-admin` (org administration), `builder` (default member) - a deliberate change from today's role set, stated as such. Scopes are re-derived from those three roles: `super-admin` and `org-admin` carry the admin scope set (`admin auth:read auth:write agent:execute agent:read agent:write`), `builder` carries the member set (`auth:read agent:execute agent:read`); platform-versus-org authority is decided by the `role` claim and the store re-resolution below, not by a distinct scope string (carried scope vocabulary; reference/invisible-behaviors.md section 1.3). Role checks on privileged routes re-resolve the user from the store, never trusting stale token claims alone, and resolve a store `admin` to `org-admin` for org-scoped resources and to `super-admin` for platform-wide operations (user creation across orgs, org creation, featured-gallery curation, global demos) (carried). The founder's account is seeded `super-admin` at migration (Amendment 2).

**Pre-auth exemptions** are exactly the `public` class of chapter 03 section 3.2: `POST /auth/login`, `POST /auth/device`, `POST /auth/device/poll`, `GET /health`, `/api/demos*`, and the public static asset routes. Device login carries its anti-abuse properties verbatim: 10-minute code TTL, 5 s poll pacing with `slow_down`, single-use approval consumed atomically at poll time, approval binds only to the approver's own account, and unknown or consumed codes report `expired` so existence is never leaked (reference/invisible-behaviors.md section 1.4). The device pending store stays in-memory by design (single process, invariant 5; reference/data-inventory.md section 9).

**App-SSO isolation.** The end-user identity plane for served apps is fully separate from platform JWTs and is carried wholesale with every security property named in reference/invisible-behaviors.md section 1.8: appId scoping enforced server-side on the session record (never via cookie path alone), appId charset guard against cookie injection, atomic single-use consumption of the OAuth `state` (find-and-delete under one transaction - chapter 04 section 4.3.3), PKCE + nonce validation on the Microsoft OIDC flow, timing-safe dummy-hash compares on password login so user enumeration is not possible, and set-password privilege checks resolved against the **session's** login collection, never the request's, so a planted row cannot forge a role. Delegated Graph tokens are stored AES-encrypted on the session record and never sent to the app; only the server-side `/api/app-sso/m365` proxy injects them.

One property in that list is strengthened on port: **pinning of the Microsoft OIDC redirect URI**. Today, when `MICROSOFT_SSO_REDIRECT_URI` is unset, the OAuth callback URI is derived from `X-Forwarded-*` headers - spoofable request input, with the Azure app-registration allowlist as the only backstop - and a production-like boot with Microsoft SSO configured only logs a loud warning (reference/invisible-behaviors.md section 1.8, redirect-URI pinning warning). The rebuild promotes that warning to a fail-closed boot gate, stated here as a deliberate change vs today: in a production-like environment with Microsoft SSO configured (client id and secret present), `MICROSOFT_SSO_REDIRECT_URI` must be set, and the pinned value is used verbatim in the start and callback flows. Header-derived fallback survives only in non-production-like environments (local dev behind changing ports). Rationale: a redirect URI assembled from attacker-influenced headers should never sit one misconfigured allowlist away from authorization-code redirection, and the platform does not control the Azure allowlist. The gate appears in the 9.7 table; the pinned URI belongs on the external URL continuity list of the cutover checklist (chapter 10 section 10.6) alongside the platform-integration console URIs, since it must also be registered in the Azure app registration.

**Where enforced:** `auth/` (JWT primitives + middleware, single mint point), `bridge/` (bridge token mint/verify), `integrations/` with the served-app routes (app-SSO plane). Mechanisms: middleware on every `/api/v1` route; unit tests for the cross-class rejection (a bridge token on a REST route must 401, a session JWT on the bridge upgrade must be refused); contract tests for the exemption list (route census: every route not in the `public`/`HMAC`/`header-scoped` classes rejects an unauthenticated request).

### Invariant 1 - No Anthropic access outside the chokepoint (FIXED-3, FIXED-8)

**What.** Every Anthropic byte in the service flows through `llm/` - the Agent SDK invocation, the direct Messages REST client, and the gateway pass-through. No other module imports Anthropic SDKs or calls Anthropic endpoints, and no model call exists in runtime platform paths (routing, lifecycle, CRUD) - the platform-attributed call sites are eliminated or made deterministic per their recorded fates (reference/llm-usage-map.md section 7; chapter 06).

**Enforced where.** Lint rule: ESLint `no-restricted-imports` banning `@anthropic-ai/*` everywhere in `api/src/**` except `api/src/llm/**` (chapter 02 section 2.9). Grep gate: CI fails if `api.anthropic.com` or `@anthropic-ai/` appears outside `api/src/llm/` (catches raw `fetch`). Module direction lint: only the modules whose chapter-02 import lists include `llm/` may import it. Test gate: chapter 06 acceptance criteria require every call site to carry an attribution tag.

**Changes vs today.** Today the SDK adapter, a direct client, the gateway, and one self-metering surface all talk to Anthropic from different modules; the rebuild collapses them into one module with one meter (chapter 06). The lint + grep enforcement is new - today the rule exists only as prose.

### Invariant 2 - Egress controls: anonymisation before Anthropic, error sanitisation to clients (FIXED-8 as amended, FIXED-13)

**What.** Two distinct egress controls guard the boundary the process sits on, and they must not be conflated:

- **Model-bound anonymisation (the primary control, amended to build scope).** Every model-bound payload in the service, from every origin - web-UI chat, integration-sourced content, bridge provider requests from the local Pi loop and the TUI, and Cortex's own internal/platform AI calls - passes the anonymisation service before it reaches Anthropic; detected sensitive spans cross only as deterministic, format-preserving tokens (invariant I3 of the Ekoa Local v2 brief, docs/, A1). This is no longer a seam: **the anonymisation layer is built in this run as part of the egress module** (FIXED-8 as amended). Chapter 17 owns the full specification - pipeline, detection layers, per-session vault, audit, service interface, and test harness; this chapter states the invariant and its enforcement homes only.
- **Client-bound error sanitisation (the carried response-side control).** No provider identity or provider-auth text ever reaches a user. This is the pure error-sanitizer applied at the SSE egress point with a client-side twin; it encodes a production incident and carries verbatim (reference/carryover-audit.md A7).

The two controls run at opposite ends of a request: anonymisation on the way OUT to Anthropic (the model-bound payload, sensitive spans replaced with tokens), error sanitisation on the way BACK to the client (outbound error text replaced wholesale). Neither substitutes for the other; diagram 10 renders the two trust boundaries the anonymisation control protects.

**Conflict recorded, not resolved silently:** there is NO Presidio integration in the existing code - a grep for `presidio` across the old source returns zero hits (reference/carryover-audit.md A7), and earlier doc claims of a live Presidio anonymiser were wrong. The rebuild closes the gap by **building the layer**, not by preserving an empty seam. Presidio may still be adopted, but only as one acceptable orchestration vehicle for the detection recognizers behind the service interface (the Ekoa Local v2 brief, docs/, A3.3); the choice is hidden behind `anonymize`/`deanonymize` (chapter 17 section 17.7) and is not load-bearing here.

**Anonymisation - enforced where.** `llm/` (`api/src/llm/`) is the one egress module and the sole owner of the Anthropic client (FIXED-13, invariant 1). The anonymisation pipeline is one of its three concerns - attribution/metering (chapter 06), anonymisation (chapter 17), and provider routing config - so a model-bound payload structurally cannot reach Anthropic without passing it. Mechanisms: lint/dependency rule (nothing outside `llm/` imports or instantiates the Anthropic client - invariant 1's `no-restricted-imports` plus the grep gate) and subprocess base-URL routing (Agent SDK spawns are pointed at the chokepoint via base URL/env so their traffic funnels through it, never around it - FIXED-13); test gate: the chapter 17 payload-capture harness (chapter 17 section 17.8) asserts that a planted synthetic structured ID and a deny-listed party name appear tokenized, never cleartext, in EVERY captured outbound Anthropic request across all origins including TUI and bridge sessions, while the user-visible response is cleartext and a tool_use round trip resolves against the real value. No bypass path exists; this is the structural form of invariant I3.

**Error sanitisation - enforced where.** `services/` owns one function, `sanitizeOutbound(text): string`. It is applied at exactly two egress points: (a) the SSE event serializer in `events/` sanitizes every `error`-class event payload before framing (carried placement; reference/carryover-audit.md A7); (b) the Express error middleware that shapes the chapter-03 error envelope sanitizes `error.message` before responding (chapter 03 section 3.3). The sanitizer is a substring-marker scanner: outbound error text matching any provider-identifying or provider-auth marker is replaced wholesale with a brand-safe, user-actionable message - never partially redacted, since partial redaction leaks context. The marker set includes the auth-error and transient-provider strings the model can return AS result text, which the run pipeline also uses to keep provider errors out of persisted chat turns (reference/invisible-behaviors.md section 2.7). The migrated web client keeps its twin (`sanitize-error` equivalent) as defence in depth (reference/carryover-audit.md A7: "keep both layers"). Mechanisms: single write path (only the SSE serializer and the error middleware produce outbound error text; a direct `res.status(...).json({error...})` outside the error middleware is grep-banned), plus a test gate injecting a provider-auth error string into a failing run and asserting the wire output contains neither "Anthropic" nor "Claude" nor auth markers, on both the SSE and REST paths.

**Changes vs today.** (a) The anonymisation control moves from an empty "integration point" to a built layer (chapter 17); its structural enforcement (FIXED-13) is new. (b) Error-sanitisation coverage widens from SSE-only to SSE + REST error envelope (today non-SSE surfaces could leak; the new error middleware closes that). Related storage-side control, distinct from both egress controls: the memory privacy scrub (12 regex classes for tokens, keys, passwords, PII) carries with the memory extraction feature, which under P-12 as re-resolved by Amendment 2 now ships **ON** by default - automatic post-run extraction, attributed billable `user_work`, always writing `visibility: 'private'`; the scrub patterns are kept (reference/invisible-behaviors.md section 11.2; chapter 05, chapter 06).

### Invariant 3 - Single audit write path (Registo-ready) (FIXED-8)

**What.** All activity/audit logging flows through one write function; multiple read surfaces can be added later without touching writers. Audit writes are best-effort: a persistence failure is logged and swallowed, never failing the domain action (carried; reference/invisible-behaviors.md section 10.1).

**Enforced where.** One exported `logActivity(user, category, type, description, metadata?)` in `data/` - the activity store's single exported write function (chapter 02 section 2.6 homes it there deliberately: `data/` sits at the bottom of the import graph, so every writing module - `auth/`, `memory/`, `agents/`, `apps/`, `integrations/`, `knowledge/`, `billing/`, `services/` - can reach it within its chapter 02 import list). Mechanisms: single write path (no other code may write the activity collection - grep gate on the store name), review gate stated in the new repo `CLAUDE.md`. The write-site inventory of reference/invisible-behaviors.md section 10.2 is the porting checklist: every listed category/type pair must have a call site in the rebuilt domain modules (test gate: an audit-coverage test drives one operation per category and asserts a row). The categories, mapped to their new homes:

| Category | Representative types (full list in reference/invisible-behaviors.md section 10.2) | New home |
|---|---|---|
| auth | login, create-user, change-password, reset-password, device-approve/deny/login, delete-user | `auth/` + users routes |
| execute | build job created, commit-blocked (secret findings attached), job cancelled | `agents/` |
| integrations | config create/update/delete, definition delete/save, session capture, execution | `integrations/` |
| memory | create/update/delete/bulk-delete, signals | `memory/` |
| settings / branding / billing | CRUD and admin toggles | their owning services (`services/` for settings, branding; `billing/` for billing) - never the routers, which may not import `data/` (chapter 02 section 2.6); teams deleted end to end (Amendment 2) |
| app-data | backup snapshot/restore (PT-PT descriptions carried) | `data/` backups service |
| knowledge | ingest, delete, source CRUD, refresh, reindex | `knowledge/` |
| platform-integrations | connect, callback, disconnect | `integrations/` |
| artifacts | create/update/delete, file writes, version restore, fork, import, bundle/featured updates | `apps/` |
| pipedream | account/config operations | `integrations/` |

**Changes vs today.** (a) **The username bug is fixed on port**: today every row's `username` field actually stores the user id (verified; reference/invisible-behaviors.md section 10.1 and Conflicts #20). The new record carries both `userId` and the real `username`, resolved at write time. (b) Retention: 12 months per P-09 (chapter 04 section 4.6) replaces unbounded growth. (c) **Superseded by Amendment 2**: the minimal org-scoped Registo read surface is un-deferred (chapter 03 section 3.8.24). Org admins read their own org's activity metadata (who built/ran/logged what, when, usage per user - never chat/message bodies); super-admin reads across orgs; every read is itself access-logged through this same write path (section 9.8 C6). Activity rows therefore gain `orgId` (chapter 04; index `{orgId, timestamp}`). The write path stays the single Registo-ready path - this read surface is one additive reader, and content-level oversight remains an explicit future decision, not a default. (d) The second audit surface - `webhook_audit` rows for every webhook ingress outcome - is carried (invariant 9), **and the Adobe Sign ingress gap is closed on port**: today the Adobe webhook route writes no audit row on any outcome (verified; reference/invisible-behaviors.md section 10.3); the rebuilt route writes the same outcome rows as the generic ingress while keeping its immediate-200 response contract.

### Invariant 4 - Centrally managed model credentials; two auth modes; no per-user ad hoc keys; no `~/.claude` fallback (FIXED-8 as amended)

**What.** Model access authenticates exclusively via **centrally managed** credentials - encrypted at rest, never per-user ad hoc, never a fallback to a developer's `~/.claude` login. The chokepoint supports **two auth modes as per-environment configuration** (FIXED-8 as amended, founder, 2026-07-06 (amendment 2, consolidated ledger, docs/ekoa-code-spec-amendment-2-consolidated-ledger.md)): `oauth` (Claude subscription OAuth, the current mode) and `api-key` (Anthropic API key, the production/customer path). Mode is per-environment config on the egress module (FIXED-13 already owns provider routing). The credential lives in the `credentials` Firestore singleton (`_id: 'default'`, carrying `mode: 'oauth' | 'api-key'`; P-08 re-resolved - Supabase retired; chapter 04 section 4.5 as rewritten), AES-encrypted via the one crypto module (invariant 6). One credential per environment: no per-installation rows, no pools.

**Enforced where.** `llm/` (`api/src/llm/`) owns credential custody. The machinery is **simplified** from today: one credential per environment; proactive refresh before expiry in `oauth` mode; refresh-and-retry-once on a 401; an alert on persistent failure (surfaced on `/health` `claudeAuth`). The old multi-subscription rollover model is **deleted from the spec - the eight load-bearing rotation semantics are superseded**: no credential pools, no rotation mutex, no persist-first ordering, no 20-minute watchdog with peer adoption, no keep-row-on-401, no health scoring, no selection logic, no per-installation rows (the watchdog reduces to the persistent-failure alert). The SDK subprocess environment builder in `llm/` deletes any inherited provider env (`ANTHROPIC_API_KEY`/`ANTH_API_KEY`/`ANTHROPIC_BASE_URL`) and injects exactly one credential from central custody - `CLAUDE_CODE_OAUTH_TOKEN` in `oauth` mode, `ANTHROPIC_API_KEY` in `api-key` mode - then sets `ANTHROPIC_BASE_URL` to the chokepoint (FIXED-13 unchanged; reference/invisible-behaviors.md section 2.7). `agents/` configures the SDK with empty setting sources so nothing is inherited from the host machine's profile. The OAuth model-access asymmetry (direct Messages REST is FAST-only) applies in `oauth` mode only; `api-key` mode has no such asymmetry, but the chokepoint's conservative typing (`completeFast` FAST-only) is kept so the design holds identically under both modes. Mechanisms: grep gate - no provider credential env name (`ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`) appears outside the `api/src/llm/` custody/env-builder code; test gates - the credential-custody tests (proactive refresh, refresh-and-retry-once on 401, persistent-failure alert; and that each mode injects the correct single credential with no other provider variable leaking into the SDK env). Boot posture: a missing credential row logs an error and raises the persistent-failure alert; it is **non-fatal** - agent calls fail until the credential is healed (section 9.7).

**Changes vs today.** Substantial, by Amendment 2. (a) Two auth modes replace OAuth-only: `api-key` is added as a per-environment configuration for the production/customer path, `oauth` stays the current mode (FIXED-8 as amended). (b) The rollover machinery is deleted - the eight rotation semantics that each encoded a fixed production incident are superseded by the one-credential-per-environment model, since with a single credential and no peer process contending for it those incidents no longer apply. (c) The boot gate is gone: a missing credential is non-fatal (an alert, not a refusal). (d) Custody moves inside the chokepoint module, and the backing store is the `credentials` Firestore singleton, not Supabase (P-08 re-resolved). The "no per-user ad hoc keys, no `~/.claude` fallback" property is carried unchanged.

### Invariant 5 - Org and user scoping on every data access; single multi-tenant process (FIXED-8; FIXED-14 structural scoping)

**What.** The tenant is the **org** (Amendment 2, Part 4; FIXED-14 structural scoping). One process serves multiple orgs; org + user scoping is structural in the repository layer - an unscoped-by-org query is inexpressible, and cross-org access is impossible by construction. User-level scoping via `ownerUserId`/`userId` fields is enforced in the data layer under the org scope, with **uniform not-found on ownership mismatch** (existence is never revealed). Sharing runs on a `visibility: private | org` field on memories and artifacts (Amendment 2, Part 4.12): a private item is owner-only and invisible to org admins too (its existence appears in the Registo metadata surface, never its content); an org-shared artifact is visible and editable by org members (safe because git version snapshots + restore + Registo cover every mutation); the resolver injects the owner's own memories plus org-shared ones. App-data scoping stays `(appId, collection)` through a single query-binding point - the org model adds no dimension to the collections engine; the shared-owner scope `usr.<ownerUserId>` is resolved server-side only, and client-supplied ids with the `usr.` prefix are rejected (chapter 04 sections 4.2.5-4.2.6; reference/data-inventory.md section 3.3).

**Enforced where.** `data/` exclusively: scope-resolution helpers are the only query constructors and require the caller's org + user context; routes never import `data/` (lint rule, chapter 02 section 2.9), so every access goes through a domain module that passes that context down. The ownership gate on integration credential use is carried verbatim: failure masks as not-found (reference/invisible-behaviors.md section 3.2). Mechanisms: data-layer helper + lint rule + test gate - the carried parity-test discipline for the app-data plane (chapter 04 section 4.2.8), per-store ownership tests (user B reading user A's resource gets 404, not 403), the **cross-org adversarial suite** (an authenticated user of org A attempts every operation against org B's brand, knowledge, memories, artifacts, automations, and Registo, and gets a clean 403/404 only - chapter 13), and the **in-org sharing tests** (builder A reading builder B's private memory gets 404; builder A editing builder B's private artifact gets 403; an org-shared artifact is editable by an org member).

**Changes vs today.** (a) **Tenancy is now the org** (Amendment 2): the old single-company install becomes a multi-org single process, with the org as the structural scope on branding, the knowledge base, integration credentials, anonymisation deny-lists, org-shared memories/artifacts, automations, and the Registo read surface; the former per-company scoping is superseded (the historical "single-tenant install" rationale in the registers stays as history). (b) The single-process assumption becomes an explicit, stated commitment rather than an accident: the in-memory single-instance state inventory (device codes, SSE replay buffers, slug index, run registries, daemon bridge registry, and the activation map of section 9.7.1 - reference/data-inventory.md section 9) is legal because FIXED-8 fixes one process; the spec forbids horizontal scaling without revisiting that list. (c) The old filesystem-path isolation that gave scoping "for free" is replaced by the explicit single query-binding point (chapter 04 section 4.2.6).

### Invariant 6 - Credential encryption at rest; key mandatory; single crypto module (FIXED-8)

**What.** Integration credentials, trigger HMAC secrets, captured browser session state, and delegated Graph tokens are AES-256-GCM ciphertexts at rest, decrypted only just-in-time at execution, never exposed to any LLM context or client response (chapter 04 section 4.7). One deliberate, carried exception is recorded honestly: the integration execution path returns decrypted credentials to the **authorized owner** over the API after the ownership gate - the "model never sees credentials" property holds everywhere including there; the "client never sees credentials" property has that one owner-gated exception (reference/invisible-behaviors.md section 3.2).

**Enforced where.** `data/` owns THE crypto module - the only AES implementation in the repository. Mechanisms: boot gate (`ENCRYPTION_KEY` absent = refuse to boot, every environment, chapter 04 section 4.7), grep gate (no second `createCipheriv` call site outside the crypto module; the old default key constant must not appear anywhere), test gate (round-trip tests plus a fixture decrypting a today-formatted ciphertext to prove wire-format compatibility for migration - chapter 04 section 4.8).

**Changes vs today.** Three fixes on port. (a) **Single module**: today two parallel AES-256-GCM implementations exist with the same format and key derivation (verified; reference/invisible-behaviors.md section 3.1 and Conflicts #4) - unified. (b) **Mandatory key**: today the default dev key `'default-dev-encryption-key-32ch!'` has no production guard, unlike the JWT secret (Conflicts #5) - see invariant 11. (c) Key resolution is isolated behind one function so KMS can land later without touching call sites. **RESOLVED (P-14)** (owned by chapter 04 section 4.7): the single env key is kept in v1; GCP KMS envelope encryption is deferred, with the single key-resolution function as the isolated seam where KMS lands later. Resolved: defaulted to recommendation per amendment Part 1, founder, 2026-07-06 (amendment brief, docs/ekoa-code-spec-amendment-brief.md). Rejected alternative: GCP KMS envelope encryption from day one.

### Invariant 7 - Secret guard on code egress (FIXED-8)

**What.** User-app code leaves the platform through exactly three doors - version snapshot commits, the GitHub mirror push, and the code download zip - and every door runs the secret scanner. A detected credential blocks the operation loudly: the snapshot commit is refused with a `commit-blocked` audit row including findings (reference/invisible-behaviors.md section 10.2), and the download returns `422 SECRET_GUARD_BLOCKED` with distinct client copy (chapter 03 section 3.3; reference/operations-inventory.md section 8).

**Enforced where.** The pure commit-guard module ports as-is into `services/` (reference/carryover-audit.md A7); the git pipeline (`apps/` version snapshots, GitHub backup) and the archive/download path call it before writing anything outward (reference/carryover-audit.md B18 and the `app-archive` Services-sweep row). Mechanisms: single write path (all outbound git/zip operations route through the guarded pipeline functions; raw `git push`/archive calls elsewhere are review-gated), test gate: a fixture project containing a planted key must fail the snapshot with the audit row and fail the download with 422.

**Changes vs today.** None functional - carried verbatim, including the deliberate refusal to reuse the memory anonymizer for this purpose (it false-positives on hashes; the dead anonymizer module is not carried - reference/invisible-behaviors.md section 11.7).

### Invariant 8 - SSRF guard on platform-initiated fetches of user-supplied URLs (FIXED-8)

**What.** Whenever the platform itself fetches a URL that a user typed (brand research target, knowledge crawl sources and seed lists, uploaded link fetches for chat context), the URL passes the SSRF guard first: scheme allowlist, private/link-local/loopback address rejection. The knowledge source seed lists are SSRF-validated at write time as well (reference/data-inventory.md section 5.2, sources row).

**Enforced where.** The zero-dependency guard ports as-is into `services/` (reference/carryover-audit.md A11, `url-safety` row) together with the url-fetcher that already wraps it (Services sweep: url-fetcher depends on url-safety ONLY). Mechanisms: single write path - platform fetches of user-supplied URLs go through the guarded fetcher; review gate: a route- or service-level `fetch` whose URL derives from request input and does not use the guarded fetcher is a review-rejectable finding, stated in the new repo `CLAUDE.md`. Test gate: guard unit tests (localhost, 169.254.x, RFC-1918, `file:` scheme all rejected) plus one integration test per entry point (brand research and knowledge source creation reject a private-address URL with `400 VALIDATION_FAILED`).

**Scope statement (honest boundary).** User-defined integration actions call arbitrary user-configured endpoints by design - that is their purpose, they run under the owner's own credentials, and they are not SSRF-gated today. The rebuild carries that boundary unchanged and records it here so it is a decision, not an oversight.

**Changes vs today.** None functional; the review-gate convention and the per-entry-point tests are new.

### Invariant 9 - Webhook HMAC verification, dedup, and audit (FIXED-8)

**What.** The `/hooks/:triggerId` ingress (path kept verbatim - URLs are registered with providers, chapter 03 section 3.8.17) carries its whole security pipeline (reference/invisible-behaviors.md section 12.2):

1. Raw-body HMAC verification with the provider's algorithm (pure verifier module), secret stored AES-encrypted and decrypted only at verify time.
2. **Disabled-check AFTER signature**: a valid signature on a disabled endpoint returns 410, an invalid one 401 - deliberate ordering that also lets the boot self-test probe the wire path.
3. Dedup: durable enqueue with `UNIQUE(trigger_id, dedup_key)`; a collision returns 200 `{duplicate: true}` (idempotency is the uniqueness constraint, nothing else - reference/carryover-audit.md A11, event-queue row).
4. Audit: every ingress attempt writes a `webhook_audit` row with its outcome (`accepted | duplicate | rejected_signature | rejected_unknown_trigger | rejected_disabled | rejected_other`) (reference/invisible-behaviors.md section 10.3).
5. Clock-skew sampling from the `Date` header into the `/health.clockSkewSec` median - replay-tolerance windows depend on it; ops alerting keys off it (reference/invisible-behaviors.md section 6).
6. The boot-time raw-body self-test: a signed synthetic payload posted to a disabled endpoint asserts the verifier sees unmodified bytes, catching accidental JSON body-parser reordering above the ingress mount (reference/invisible-behaviors.md section 5.1).
7. Timing-safe compares on the GET hub-challenge handshake (Meta-style) (reference/invisible-behaviors.md section 12.2).

**Enforced where.** `events/` (ingress routes, verifiers, SQLite queue, audit table). Mechanisms: middleware ordering fixed in the composition root with the raw-body mount below any JSON parser; test gates: the self-test itself (kept, with its kill-switch env), verifier unit tests per provider algorithm, a dedup collision test, and an audit-row-per-outcome test.

**Changes vs today.** The Adobe Sign ingress joins the audit surface (invariant 3, change d). Its other deliberate deviations are carried as-is and recorded: credential-free public route, immediate 200 + client-id echo before async processing, authenticity re-established in the service by owner-scoped refetch of the agreement (reference/invisible-behaviors.md section 12.2).

### Invariant 10 - Sandbox path confinement; P-15 for the automation file operations (FIXED-8)

**What.** Every filesystem path derived from user input resolves through a symlink-hardened safe-path helper that jails it to the owner's sandbox (`~/.ekoa/sandboxes/user-<id>/...` today; the equivalent workspace root in the rebuild). This is how tenancy is enforced on workspace state (reference/data-inventory.md section 7.3: the `resolveSafePath` class of checks in artifact file access and the git tooling).

**Enforced where.** One safe-path helper in `services/` (chapter 02 section 2.6 homes it there so that both its consumers can import it: `apps/` for the artifact file routes, the git/versions pipeline, and the archive/download path, and `automation/` for the P-15 file operations - `automation/` may not import `apps/`). The new API additionally narrows the input surface: file paths in the artifact files endpoints are project-relative, never sandbox-absolute (chapter 03 section 3.8.9). Mechanisms: single write path (the helper is the only path-resolution primitive for user-derived paths - grep gate on `path.resolve`/`fs.` usage against request-derived values outside it is a review gate; the helper itself is unit-tested), test gate: traversal fixtures (`..`, absolute paths, symlink escape) must all fail with uniform not-found.

**RESOLVED (P-15) - confinement of the automation manifest file operations.** The automation vocabulary that user apps and the coding agent author includes `file.read`/`file.write` operations that today read and write **arbitrary filesystem paths**: the path resolver only expands `~` and otherwise returns absolute paths unchanged, the `userId` parameter is ignored, and relative paths resolve against the operator's home directory (verified; reference/data-inventory.md section 7.5). The old code's own comment concedes the gap ("reject absolute paths outside the user's sandbox? For now: trust user-issued paths"). That trust-all-paths behavior does not carry - porting it silently is explicitly forbidden (reference/data-inventory.md section 10.11).

**Decision: both operations are jailed to the owner's sandbox** using the same safe-path helper as above (mirroring the artifact-files confinement), with relative paths resolving against the owner's sandbox root and absolute paths outside it rejected with uniform not-found. The automation `file.read`/`file.write` operations therefore pass the same traversal/symlink/absolute fixtures as the artifact-files path. Manifests that legitimately need a shared location get an explicit, admin-approved allowlisted root as a follow-up feature, not a silent bypass. Resolved: ACCEPT (recommendation final), founder, 2026-07-06 (amendment brief, docs/ekoa-code-spec-amendment-brief.md). Rejected alternative: drop `file.read`/`file.write` from the vocabulary entirely and route automation file state through the collections engine (FIXED-5) - cleaner, but it breaks existing manifests and removes a legitimately useful local-file capability.

**Changes vs today.** The P-15 confinement is the change: today's unconfined behavior does not carry, and the automation file operations now jail to the owner sandbox. The artifact-files confinement itself carries unchanged, minus the sandbox-absolute path input shape (retired in chapter 03).

### Invariant 11 - Production guard on default secrets, extended to the encryption key (FIXED-8)

**What.** Today the JWT secret has a fail-closed guard: the process throws at load when the secret is unset or the dev default in a production-like environment, where production-likeness is derived from more than `NODE_ENV` because that variable is unset on the deploy path (reference/invisible-behaviors.md section 1.1). The encryption key has **no such guard** - the asymmetry is recorded as a conflict (reference/invisible-behaviors.md Conflicts #5).

**Enforced where.** `config.ts` boot validation: (a) JWT secret guard carried as-is, including the production-likeness derivation; (b) `ENCRYPTION_KEY` goes further than parity - it is mandatory in **every** environment with no default constant existing anywhere in the repository (chapter 04 section 4.7); the dev bootstrap script generates one into the local env file. Mechanisms: boot gate + grep gate (the old default key literal, and any new hardcoded key, fail CI) + test gate (boot-without-key test asserts refusal; chapter 04 acceptance criterion 6).

**Changes vs today.** The encryption-key guard is new and stricter than the JWT guard (mandatory everywhere, not just production-like). No key-rotation mechanism exists today (a changed key silently orphans ciphertexts - reference/invisible-behaviors.md section 3.1); v1 does not add rotation, but the single key-resolution function (invariant 6) is the seam where rotation/KMS lands (P-14).

## 9.4 Credential-injection planes and their gates

Three server-side planes inject workspace or session credentials into requests on behalf of served apps, so the page never holds a token (reference/invisible-behaviors.md sections 1.2, 1.8):

| Plane | Credential injected | Gate today | Gate in rebuild |
|---|---|---|---|
| `/api/integration/:key/*` proxy | The key's decrypted integration credential | optional-JWT (invalid 401, absent proceeds); key-scoped by path | carried (chapter 03 section 3.9); key scoping is the boundary |
| `/api/app-sso/m365/*` | The end user's delegated Graph tokens from their app-SSO session | valid per-app SSO cookie required | carried verbatim (section 9.2) |
| `/api/m365/*` workspace Graph proxy | The org's WORKSPACE Microsoft platform token (per-org custody, Amendment 2 Part 4.9) | optional-JWT only; **no per-app allowlist or header gate is enforced in the route, despite an inline comment claiming X-Ekoa-App-Id scoping** (verified; reference/invisible-behaviors.md section 1.2) | **RESOLVED (Q-10)**: `X-Ekoa-App-Id` verified + per-app manifest opt-in (below) |

**RESOLVED (Q-10) - gate the workspace Graph proxy.** Today an unauthenticated same-origin request can reach arbitrary caller-chosen Graph paths with the workspace token injected; the comment-claimed scoping does not exist in the code (verified; reference/invisible-behaviors.md section 1.2). The rebuild gates it normatively: every `/api/m365/*` request must carry and pass verification of `X-Ekoa-App-Id` - slug-resolved, charset-checked, and the app must exist and be served - mirroring the app-data plane gate, AND the resolved app's manifest must carry a per-app opt-in flag for the workspace Graph proxy. A request missing the header, failing slug resolution, or resolving to an app without the opt-in flag is rejected before the token is injected. Mechanisms: middleware on the `/api/m365/*` mount (header presence + slug resolution + charset guard + manifest-flag check, all ahead of token injection) and a test gate (a request without the header, and a request from an app lacking the opt-in flag, are both refused; a request from an opted-in served app with a valid header succeeds). The workspace Microsoft token this proxy injects is per-org custody under Amendment 2 (Part 4.9 - integration credentials are org-scoped); the Q-10 gate itself is unchanged. This is a behavior change on an otherwise byte-compatible served-app plane (chapter 03 section 3.9); the sweep of which existing served apps call the proxy without the header - and therefore break when the gate flips on - is a named cutover checklist item (chapter 10). Rejected alternative: carry the open proxy as-is and accept that any page served from the app origin can act as the workspace account. ## 9.5 JWT-in-URL surfaces (Q-05)

Three surfaces put a bearer credential in a URL; each is justified and bounded, or transitional:

1. **SSE `?token=`** on the four sanctioned streams (chapter 03 section 3.6): carried, because EventSource cannot set headers (CONV-1). Bounded by token TTL and by the log-redaction rule below.
2. **App preview `?token=`** for owner-checked non-shareable artifacts (cross-origin dev iframes cannot share cookies; deliberately omitted for shareable artifacts) - **RESOLVED (Q-05)**: today's query-token behavior is carried, bounded by the log-redaction rule below rather than moved to same-origin session cookies. The residual tradeoff (tokens can leak into logs, browser history, referrers) is accepted and mitigated by token TTL plus redaction; the request-logging middleware isolates the surface so a later move to cookies needs no route changes (reference/frontend-cleanup-audit.md FC-024, FC-068; chapter 03 section 3.2). Resolved: defaulted to recommendation per amendment Part 1 (carry `?token=` + log-redaction middleware), founder, 2026-07-06 (amendment brief, docs/ekoa-code-spec-amendment-brief.md).
3. **Bridge WS upgrade `?token=` fallback**: transition-only, slated for removal; Bearer-header auth is the target (chapter 03 section 3.10).

**Log redaction rule (new, this chapter).** The request-logging middleware (composition root, chapter 02) must redact the `token` query parameter before any log or audit write, on every surface. Mechanism: middleware + a test gate asserting a logged SSE request contains `token=[redacted]`. This bounds the exposure of surfaces 1-3 regardless of how Q-05 resolves.

## 9.6 Token revocation (P-03)

**RESOLVED (P-03) - server-side revocation list.** Platform JWTs remain stateless bearer tokens minted by the single mint point (section 9.2), with the explicit `POST /auth/refresh` endpoint kept and `GET /auth/me` token-free; but the rebuild adds a server-side kill switch, resolving the former no-revocation tradeoff. The accepted-risk posture of today - a leaked `rememberMe` or device token valid for up to 30 days and unkillable server-side (reference/invisible-behaviors.md section 1.3 and Conflicts #6) - does not carry. Resolved: ALTERNATIVE (server-side revocation list), founder, 2026-07-06 (amendment brief, docs/ekoa-code-spec-amendment-brief.md).

**Mechanism.** An in-memory revoked-token set is the hot-path structure, backed by a small persisted `revoked_tokens` collection (chapter 04 section 4.3.1: one row per revoked token carrying the token jti/hash, `userId`, and the token expiry, the expiry driving self-pruning so the set never grows past live tokens). The set is loaded from the collection at boot and consulted in the auth middleware on every authenticated request: a token whose jti/hash is in the set is rejected exactly as an expired token is, before any route logic runs. Correct under FIXED-8 (single process): the in-memory set is authoritative for the running process, and the collection makes it survive restart. Cost is O(1) in-memory per request; the collection is read once at boot, not per request. The boot load rides on the storage backend being up, which is already a fatal boot gate (section 9.7), so an unreadable store never leaves the process serving with an unknown revocation state.

**Logout.** `POST /auth/logout` (chapter 03) revokes the caller's current token by inserting its jti/hash into both the in-memory set and the collection; an admin variant revokes another user's tokens, enabling operator "log out user X". Refresh is independent: `POST /auth/refresh` mints a new token, and revoking the old one does not touch the new.

**Enforced where.** `auth/` (the middleware check point and the mint/refresh/logout functions), `data/` (the `revoked_tokens` collection and its single write path), and the composition root (boot-time load of the set). Mechanisms: **middleware** (the per-request revocation check runs inside the same auth middleware that verifies the JWT, so no route can opt out) + boot load + **test gate** (a token that has been logged out is rejected on its next request with the same status as an expired token; an admin logout of user X invalidates user X's outstanding tokens; the set repopulates correctly from the collection across a simulated restart). The middleware placement means the check point is exactly the isolation seam the earlier no-revocation design had reserved for this addition, so it needs no route changes.

### 9.7.1 Activation gates (Amendment 2)

Amendment 2 (Part 3) replaces licensing with an activation model (founder, 2026-07-06 (amendment 2, consolidated ledger, docs/ekoa-code-spec-amendment-2-consolidated-ledger.md)). Two independent facts gate access: **`users.active`** (a boolean, admin-controlled) and the existing allowance/billing gate (chapter 06). There are no tiers.

**Three admission planes** consult a cached activation state before serving a request:

1. the `/api/v1` auth middleware (the platform-JWT plane) - the caller's own activation;
2. the served-app plane gate - keyed on the **artifact owner's** activation (a deactivated owner's app stops serving/assistant/data-plane traffic);
3. the bridge pairing plane (chapter 18) - a deactivated owner's pairing refuses connect, delegation, and provider traffic.

**Cache.** An in-memory activation map with **write-through invalidation**: the toggle write (`PATCH /users/:id { active }`) updates the map synchronously in the same operation, so a deactivation is effective immediately with no TTL wait; a periodic TTL refresh is a safety net only. This is sound under FIXED-8 (single process): the in-memory map is authoritative for the running process. **Deactivation does two things in the same operation**: it sets `active=false` AND pushes the user's outstanding tokens into the P-03 revocation set (section 9.6), so a deactivated user is both refused at the activation gate and stripped of live tokens.

**Error codes** (CONV-2, chapter 03 section 3.3): `ACCOUNT_DISABLED` (403), PT-PT "A sua conta está bloqueada. Contacte o suporte."; `BILLING_LOCKED` (402), PT-PT "A sua conta tem um problema de faturação. Contacte o suporte." The `active=false` gate returns `ACCOUNT_DISABLED`; the allowance/billing gate returns `BILLING_LOCKED`.

**Mechanisms:** middleware (the activation check runs in the same auth middleware that verifies the JWT, and in the served-app and bridge plane gates - no route opts out) + write-through map (the single write path for `active` updates both the store and the map and revokes tokens) + test gate (a deactivated user is refused with `ACCOUNT_DISABLED` on all three planes, effective immediately via write-through with no TTL wait, and their tokens are revoked; a billing-locked user is refused with `BILLING_LOCKED`). Payment-provider automation is a future webhook flipping the same `active` boolean - recorded here, not built.

### 9.7.2 Persistence postures (Amendment 2)

The three-posture data policy is normative here and in chapter 17 (Part 2.5, founder, 2026-07-06 (amendment 2, consolidated ledger, docs/ekoa-code-spec-amendment-2-consolidated-ledger.md)). Every piece of persisted or transmitted content sits in exactly one posture:

1. **Egress: tokenize.** Everything model-bound crosses the chapter 17 anonymisation pipeline before it reaches Anthropic (invariant 2). This explicitly includes memories re-injected by the resolver: they are model-bound text and tokenize with the **current session's vault** automatically, like any other outbound payload - no separate path.
2. **At-rest knowledge (messages, memories, knowledge base): cleartext, protected by scoping.** Stored cleartext so the product can read it, protected by **org + user** scoping (invariant 5), encrypted at rest, enumerated in the custody map, and covered by the C4 per-org erasure workflow (section 9.8).
3. **At-rest operational surfaces (logs, run records): omit or irreversibly redact.** Log discipline: no message bodies, no delegation file paths, no tokens - identifiers, codes, and counts only, plus the section 9.5 redaction middleware. **P-27** (resolved 2026-07-06, Amendment 2; chapter 17 section 17.10 owns): executor-face content-bearing run-record fields pass through the detector **at persist time** with **irreversible redaction** of detected spans - never session tokens (the vault is ephemeral; a token written to a permanent record is permanent noise).

## 9.10 Delegation and bridge security (chapter 18)

Local file access via the ekoa-local daemon adds a delegation path and a bridge channel whose full security model - the six principles S1-S6 - is specified in chapter 18 section 18.5 and rendered in diagram 11. This chapter records only the security invariants that touch surfaces it already owns:

- **The bridge is a distinct credential plane, already fenced off from the platform JWT plane.** The platform session JWT verifier positively rejects bridge tokens - any `connectionId` claim or `aud: ekoa-bridge` fails platform verification, and the bridge verifier rejects session tokens (section 9.2, token classes). That carried cross-class rejection IS an enforcement point for chapter 18's S2 (task binding to a pairing id) and S4 (an authenticated, revocable channel): a bridge/pairing credential can never act on the REST/`/api/v1` surface, and a platform token can never open the bridge upgrade. Chapter 18 section 18.3 adds the org-scoped pairing registry and the server-side revoke-pairing kill switch on top of this.
- **The bridge transport is explicitly outside FIXED-2's frontend rule.** FIXED-2 bans WebSockets between frontend and Cortex as API transport (with the one canvas media-channel exception, Q-01); the daemon-to-Cortex bridge is a different party - one outbound WS from the daemon, TLS, pairing-token auth at connect - and chapter 18 section 18.3 states that carve-out. This chapter notes it so the two WS exceptions are not conflated.
- **The provider endpoint for bridge traffic routes through the chokepoint (FIXED-13).** Bridge completion requests are model-bound payloads and therefore pass the anonymisation service (invariant 2) with session-identity propagation, so a stolen provider credential cannot address another org's session or vault (chapter 18 section 18.4; S6 auditability joins the correlation id minted at the chokepoint). Injection carried in file content is contained by the absence of exfiltration primitives on the delegation path - no upload, an egress cap, and the ledger (chapter 18 S5) - which is the delegation-path form of the prompt-injection posture at section 9.8 B3.

Diagram 11 renders the S1-S6 bindings; the delegation and bridge detail is chapter 18's, not restated here.
