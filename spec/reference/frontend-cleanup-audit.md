# Frontend Cleanup Audit — `ekoa/` → `web/` Migration Map

This document is the migration map for moving the existing Next.js frontend (`ekoa/`) into the rebuilt repo's `web/` directory against a conventional typed-REST Cortex backend. It inventories (1) every wire-protocol touchpoint (RPC dispatch, SSE subscription, event-bus assumption) with the typed-REST-client replacement each needs, (2) dead code and UI for features that no longer exist, (3) components that assume content-driven backend behaviors (recipes, skills, orchestrator markers), and (4) visibly stale items (naming, copy, obsolete flows). Every item carries a stable id (`FC-NNN`) and a verdict — **migrate** / **clean** / **delete** — so the web-client-migration chapter of the spec can assign a fate to each one.

**Method:** derived from source code at commit `3882aa6` (HEAD at audit time). CLAUDE.md and `docs/` treated as hints only — where a doc contradicts code, the contradiction is recorded, never resolved silently. Evidence gathered by two independent finder sweeps and spot-verified against source (importer/caller greps across `app/ components/ hooks/ lib/ stores/ __tests__/ e2e/`; backend cross-checks against `cortex/src/handlers/index.ts` and `cortex/apps/`). All paths are relative to `ekoa/` unless prefixed `cortex/`. "RPC" means the current single-endpoint action protocol `POST /api/v1/action {app, intent, params, request_id}`.

**Verdict legend:**

| Verdict | Meaning |
|---|---|
| **migrate** | Port the behavior/contract into `web/` largely as-is (implementation may be rewritten, contract kept). |
| **clean** | Port the *capability* but rework it — fix the defect, rename, restructure, or make an explicit contract decision noted in the reason. |
| **delete** | Do not port. Dead, superseded, or a vestige of a removed system. |

## 1. Protocol touchpoints

Everything in this section talks to Cortex over `POST /api/v1/action` (RPC), `POST /api/v1/request` (chat runs), `GET /api/v1/events` (SSE), or a raw HTTP route. The replacement column states what the new typed REST client must offer.
